# 开发疑难杂症记录

## 案例 HB-TRBL-2026-04-10-01
### 主题
课程详情页首进时统计数据已拿到，但界面显示仍为 `000`。

### 现象
- 课程详情请求链路日志显示：最终统计值已经是 `total=4, completed=0, progress=0`。
- 页面应用数据日志也显示：`pageData` 已写入相同统计值。
- 但顶部统计卡片仍显示 `000`，与日志不一致。

### 影响范围
- 页面：`CoursePage` 课程详情统计展示区。
- 链路：`loadCourseDetailPageData -> ensureCourseSessionCache -> mergeSessionDetailWithStartSummary -> 页面应用`。

### 排查过程
1. 在服务层和页面层增加 `CourseDetailTrace` 关键节点日志，确认“接口值 -> merge 值 -> 页面应用值”全链路一致。
2. 修复日志实现中的 ArkTS 编译问题：
   - 避免把未声明类型的对象字面量直接传给日志 helper（会触发 `arkts-no-untyped-obj-literals`）。
   - 避免使用索引签名兜底日志参数（会触发 `arkts-no-indexed-signatures`）。
   - 统一改为字符串形式日志：`key=value`。
3. 页面临时增加可视化调试文本（红字）验证实时状态，确认状态本身正确。
4. 将统计卡片从“带参数 Builder”调整为“无参数 Builder，直接读取页面 `@State` 基础类型字段”。

### 根因结论
问题不在接口与数据合并链路，而在展示绑定层：
- 统计卡片原实现依赖带参 Builder + 对象态字段传值；在该场景出现渲染读取滞后，导致界面未及时反映最新值。
- 页面实际状态已经更新，但统计卡片读取到的不是最新展示态。

### 最终修复
- 在 `CoursePage` 增加基础类型展示态：
  - `displayTotalItemCount`
  - `displayCompletedItemCount`
  - `displayProgressPercent`
- 在数据加载与页面数据应用后统一调用同步方法，把 `pageData` 归一化写入上述基础类型字段。
- 统计卡片 Builder 直接读取上述 `@State` 字段，不再通过参数传递统计值。
- 同时修正服务层日志中的“是否命中 start 兜底”判定：仅 `> 0` 才记为命中，避免 `0` 造成误导。

### 验收结果
- 课程详情首进时，统计展示与日志一致。
- 构建通过，未再出现新增编译错误。
- 用户实机确认问题已解决。

### 复盘清单（后续同类问题优先执行）
1. 先用链路日志确认“值是否真的到达页面应用层”，避免在接口层盲目反复改动。
2. 当日志与 UI 不一致时，优先怀疑展示绑定与渲染路径，而不是继续追查请求层。
3. 页面关键展示值优先使用基础类型 `@State`，减少对象层级传值导致的同步不确定性。
4. ArkTS 下日志调试避免未类型化对象字面量，尽量使用明确类型或字符串日志。
