# 听桥系统面试介绍

## 一、项目整体定位

听桥系统是一个面向听障沟通训练场景的 HarmonyOS 应用，核心目标是把口语训练和手语训练结合起来，帮助用户通过课程、训练项、练习、评估、记录等完整链路持续提升表达能力。

如果面试官让我先整体介绍，我会这样说：

> 我做的是一个 HarmonyOS 端的口语与手语训练平台，项目名称叫听桥。它不是单纯的内容展示类应用，而是围绕训练闭环设计的产品，包含用户登录、课程列表、课程详情、训练项多阶段学习、练习提交、评分反馈、学习记录、收藏和个人中心等完整模块。我的工作重点主要在前端架构设计、ArkUI 页面状态组织、和后端接口联调，以及像实时语音识别、文件上传、训练评分页这种业务复杂度比较高的模块。

---

## 二、项目结构我会怎么讲

### 1. 简短版

这个项目可以概括成四条主线：

1. 用户体系：登录注册、资料编辑、修改密码、退出登录、收藏。
2. 课程体系：课程列表、搜索筛选、课程详情、训练项列表。
3. 训练体系：介绍页、视频示范、步骤动作、练习、评分结果、课程总结。
4. 数据体系：首页推荐、今日状态、学习记录、趋势统计、最近学习课程。

### 2. 稍展开版

我在做这个项目时，不是把它当成几个孤立页面来写，而是把它当成一个完整训练产品来设计：

- 首页负责提供启动后的第一印象，包括主视觉、今日推荐、今日状态和快捷入口。
- 课程页负责帮助用户找到训练内容，所以这里有搜索、分类标签、排序和隐式分页。
- 课程详情页负责承接课程信息和训练项入口。
- 训练页是项目最核心的部分，用户从课程详情点进去之后，会经历介绍、示范、步骤、练习、评分这几个环节。
- 记录页负责做训练历史的可视化展示。
- 我的页面负责用户身份、资料、收藏、账号安全等长期能力。

## 三、最值得讲的模块

## 1. 训练主链路设计

### 1.1 面试口述版

听桥项目里最核心的部分是训练主链路。我把训练过程拆成了几个明确阶段：训练介绍、示范视频、步骤动作、练习、评分结果，以及课程结束后的总结。这样的拆分方式有两个好处：一是用户路径清晰，二是前端状态管理更稳定，因为每个阶段都有清晰的输入和输出。

后面为了贴近真实训练场景，我又把逻辑做了调整：从课程详情进入任意训练项时，统一先进入介绍页，不再因为这个训练项之前完成过就直接跳评分页；评分页只在本次提交成功后展示一次。这样用户体验会更统一，也避免了老数据和本次结果混在一起。

### 1.2 关键实现

- 训练页不是单一静态页面，而是围绕“阶段状态”驱动渲染。
- 课程点击训练项之后，会先启动或恢复训练会话，再读取当前训练项详情。
- 训练项详情和训练项目录做了拆分：
  - 会话详情接口只返回训练项简表。
  - 点击某个训练项后再单独请求该题的完整详情。
- 评分页的数据不再依赖本地 mock，而是直接吃提交接口返回的 `resultDetail`。
- 为了避免后端返回值被后续刷新冲掉，我调整了提交后的状态流转逻辑，保留提交接口返回的评分结果直接进入评分页。

### 1.3 体现的能力

- 复杂业务流程拆解能力
- 多阶段页面状态管理能力
- 前后端接口边界设计能力
- 页面交互和数据生命周期控制能力

### 1.4 可能追问

#### 问：为什么要把训练项目录和训练项详情拆开？

可以回答：

因为课程训练页只需要展示训练项简表，不需要一次把每个训练项的完整内容都拉下来。这样做有三个好处：

1. 首页进入课程详情和训练目录会更快。
2. 接口结构更清晰，简表和详情职责分离。
3. 后续如果训练项内容更复杂，比如多视频、多资源、多维评分，也不会把目录接口拖得很重。

#### 问：为什么不让已完成题目直接进入评分页？

可以回答：

因为评分页本质上应该展示“本次训练提交后的即时结果”，如果把历史完成状态和本次提交结果混在一起，用户会困惑，也容易出现展示的其实是旧结果的问题。所以我把入口统一成介绍页，评分页变成一次性的结果页，这个模型更符合训练场景。

### 1.5 注意别讲错

- 不要说训练项详情一直跟着目录接口返回，现在已经拆成简表接口和单题详情接口。
- 不要说评分页是纯前端拼出来的，现在已经优先用提交接口返回的真实数据。

## 2. 实时语音识别替代音频上传

### 2.1 面试口述版

项目里口语训练最开始的方案是让用户选择本地音频再上传，但这个方案体验比较差，不像真实训练。所以后面我参考另一个最小可用项目，把口语练习改成实时语音识别：用户进入口语练习后，直接开始说话，前端实时显示识别内容，再把识别结果发给后端参与评分。这比上传本地音频更符合实际训练体验。

### 2.2 关键实现

- 接入 HarmonyOS 的语音能力，建立识别引擎生命周期。
- 训练页在口语练习模式下，初始化识别器、启动识别、停止识别、实时显示识别文本。
- 离开训练页、切换训练项、重置训练状态时，会同步销毁或停止识别引擎，避免资源泄漏。
- 提交评分时，不再依赖音频文件上传，而是把识别结果直接作为请求体字段提交给后端。

### 2.3 体现的能力

- HarmonyOS 能力接入能力
- 复杂端侧状态管理能力
- 用户体验驱动的技术重构能力

### 2.4 可能追问

#### 问：为什么要放弃本地音频上传？

可以回答：

一方面是体验问题，用户训练时不应该先录完再选文件上传；另一方面是平台能力适配问题，本地文件上传在 HarmonyOS 上的 URI 和上传能力兼容性比较复杂，而实时识别更符合口语训练本身的业务目标。

#### 问：识别结果为什么还要传给后端？

可以回答：

因为训练评分最终还是要统一走服务端的评估逻辑，前端负责采集和展示实时结果，后端负责综合判定和生成最终反馈，这样前后端职责更清晰。

### 2.5 注意别讲错

- 不要说口语训练完全不依赖后端，最终评分还是由后端给。
- 不要说语音识别是上传音频后做的，现在是端侧实时识别后再提交文本结果。

## 3. 文件上传问题排查和最终方案

### 3.1 面试口述版

这个项目里我还处理过一个很典型的跨端问题，就是 HarmonyOS 文件上传。最开始我走的是常见的 `request.uploadFile` 方案，但实际上遇到了比较多问题：相册返回的 URI 不能直接上传、上传任务创建阶段就失败、图片和视频的行为也不一致。后来我和后端一起重新梳理了上传协议，最终改成前端把文件读取成二进制，再走 HTTP 直接上传，这条链路才稳定跑通。

### 3.2 关键实现

- 前期做了最小上传验证页，把问题从业务链路里剥离出来单独排查。
- 最终图片上传改为：
  - 通过相册选择图片
  - 读取图片二进制数据
  - 用 `http` 请求直接上传到后端二进制上传接口
- 这个方案跑通后，再逐步复用到头像上传和训练提交等场景。

### 3.3 体现的能力

- 跨端问题排查能力
- 与后端共同定义接口能力
- 最小验证与逐步回收进正式业务的工程习惯

### 3.4 可能追问

#### 问：为什么不直接使用平台上传 API？

可以回答：

因为在这个项目里，平台上传 API 对相册 URI 和文件路径的兼容性不稳定，问题出在客户端任务创建阶段，还没真正进到后端。我最后选择二进制上传，是因为这条链路更可控，也更容易和后端一起统一规范。

## 4. 课程列表和搜索筛选

### 4.1 面试口述版

课程列表页我做得比较偏产品化，不只是简单拉列表。它有搜索、训练类型切换、排序、标签筛选、隐式分页、封面懒加载，而且整个筛选区的 UI 结构也是经过几轮调整的，最后形成了三行：搜索栏、训练类型和排序、横向滚动标签。

### 4.2 关键实现

- 搜索栏和筛选区做了独立的顶部布局。
- 每次只请求固定数量课程，滚动到底部自动加载下一页。
- 课程标签一开始是从列表接口动态汇总，后面根据后端接口升级，改成了独立标签接口统一返回。
- 课程卡片封面支持懒加载和占位图。

### 4.3 体现的能力

- 页面信息架构能力
- 列表性能和分页设计能力
- 与后端一起演进接口结构的能力

### 4.4 可能追问

#### 问：为什么标签后来要单独做接口？

可以回答：

因为如果标签只是从当前已加载课程里动态汇总，就不是稳定标签池，用户看到的筛选项会随分页变化，这是不合理的。单独接口可以把标签集合作为静态筛选条件一次返回，前端也更容易做统一交互。

## 5. 记录页和数据可视化

### 5.1 面试口述版

记录页我重点做了两件事：一是把原来分散的数据来源收敛成统一记录总览接口；二是把趋势图展示逻辑做了几轮调整，让它既能看趋势，也不至于把小分差展示得夸张失真。

### 5.2 关键实现

- 记录页进入时才调用记录接口，不在首页阶段无意义拉取整套数据。
- 后端后面升级成一次返回 `day/week/overview` 三套趋势数据，前端切换时不再重复请求。
- 最近学习课程单独走分页接口，并且做成隐式分页。
- 趋势图里对 0 分数据、标签文案、柱高映射做了多轮优化。

### 5.3 体现的能力

- 接口治理能力
- 数据展示和产品理解能力
- 可视化细节调优能力

### 5.4 可能追问

#### 问：为什么记录页不是每切一个 tab 就请求一次？

可以回答：

一开始是这样做的，但后面后端升级成一次返回三套趋势数据后，前端就改成单次请求、本地切换，这样请求更少，切换也更流畅。

## 6. 用户体系和完整产品闭环

### 6.1 面试口述版

除了训练主链路，我还把用户体系做成了比较完整的闭环，包括登录注册、资料编辑、修改密码、退出登录、收藏课程和训练项、最近学习、我的页面等。这部分的重点不只是把接口接上，而是保证登录态、页面可见内容和后端真实数据是一致的。

### 6.2 关键实现

- 没有用户信息就不能视为已登录，会直接退回登录页。
- 用户信息、收藏信息、首页配置、课程列表、课程详情都逐步改成真实接口驱动。
- 退出登录接上后端接口，同时清空本地 token 和登录态。
- 我的收藏支持课程和训练项两个维度。

### 6.3 体现的能力

- 用户状态一致性设计能力
- 多模块联动能力
- 后端接口变更适配能力

## 五、完整串讲版

如果面试官让我连续讲两三分钟，我会这样说：

> 听桥是一个基于 HarmonyOS 的口语与手语训练平台，我在这个项目里主要负责前端架构、核心页面实现和与后端的联调。这个项目最有代表性的地方在于它不是普通的信息展示应用，而是一个完整训练闭环。用户登录后，可以在课程列表里通过搜索、标签、排序找到课程，进入课程详情后查看训练项，再进入训练主流程。训练流程被我拆成了介绍、示范视频、步骤动作、练习、评分结果和课程总结几个阶段。  
>  
> 在实现过程中，我比较重视接口边界和状态管理，比如训练项目录和训练项详情拆成了两个接口，评分页也改成直接吃提交接口返回的结果详情，而不是前端自己拼默认数据。  
>  
> 另外一个比较有代表性的点，是我处理了 HarmonyOS 上的文件上传和实时语音识别问题。文件上传最开始踩了很多坑，最后我和后端一起把方案收敛到二进制上传；口语训练则从原来的音频上传改成了实时语音识别，让用户在训练时的体验更接近真实场景。  
>  
> 除了训练主链路，我还接入了用户登录资料、收藏、记录页趋势分析、首页推荐和今日状态这些模块，整体上把项目从一个静态演示应用，逐步推进成了一个比较完整的产品形态。
