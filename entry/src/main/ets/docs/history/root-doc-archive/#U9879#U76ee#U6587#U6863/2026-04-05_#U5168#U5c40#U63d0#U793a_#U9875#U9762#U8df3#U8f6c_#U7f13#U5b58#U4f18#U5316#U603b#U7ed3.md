# 2026-04-05 全局提示、页面跳转与缓存优化总结

## 1. 概览

今天主要完成了三块工作：

1. 全局 Toast 提示体系整理
2. 页面跳转逻辑重构
3. 课程详情与训练项内容的单次会话缓存

这三块的目标是一致的：

- 提升操作反馈的一致性
- 提升页面切换的丝滑感
- 减少重复请求，缩短重复进入页面的等待时间


## 2. 全局 Toast

### 2.1 为什么要做

之前项目里的提示比较散：

- 有的提示写在页面局部
- 有的提示没有统一样式
- 成功、失败、引导类提示没有明确区分
- 某些提示太重，某些又看不到

这样的问题是：

- 用户不容易理解当前发生了什么
- 操作成功和失败的反馈不稳定
- 视觉风格不统一


### 2.2 当前实现

全局提示现在分成了统一的组件和统一的调用入口：

- 组件：
  - [GlobalToast.ets](D:/HearBridge-User/entry/src/main/ets/pages/components/GlobalToast.ets)
- 调用：
  - [GlobalToast.ets](D:/HearBridge-User/entry/src/main/ets/api/GlobalToast.ets)

挂载位置在：

- [Index.ets](D:/HearBridge-User/entry/src/main/ets/pages/Index.ets)

这样做的意义是：

- 所有页面都能共用同一套提示
- 不需要每个页面单独实现提示 UI
- 成功、失败、引导这类语义可以统一管理


### 2.3 现在支持的提示类型

当前已经支持以下语义类型：

- `success`
- `error`
- `info`
- `warning`
- `hint`

大致使用原则：

- `success`
  - 保存成功、提交成功等明确成功结果
- `error`
  - 后端错误、网络错误、关键失败
- `info`
  - 登录成功、退出登录、密码修改成功后重新登录等轻量结果
- `warning`
  - 预留给风险或确认前提醒
- `hint`
  - 再滑一次退出应用、轻提示引导


### 2.4 当前样式策略

Toast 目前是：

- 贴近底部显示
- 单行展示
- 水平居中
- 根据语义切换颜色
- 不再强制多行宽块

同时已经针对用户反馈做了几轮收敛：

- 不再固定很宽
- 改成单行轻提示
- 提示位置统一
- 与下边栏的相对位置统一


### 2.5 已经接入的场景

目前已接入或已明确统一的场景包括：

- 资料保存成功/失败
- 头像保存成功/失败
- 头图保存成功/失败
- 修改密码成功/失败
- 退出登录成功/失败
- 发送验证码成功/失败
- 登录成功/失败
- 注册成功/失败
- 再滑一次退出应用


### 2.6 本地校验与 Toast 的边界

当前已经明确了一条原则：

- 前端本地校验错误：
  - 不用 Toast
  - 用输入区域附近的红字提示
- 后端返回错误、网络错误、关键操作结果：
  - 用全局 Toast

这样做的好处是：

- 本地校验不会频繁打断用户
- 真正需要全局感知的结果才会弹出来


## 3. 页面跳转逻辑

### 3.1 重构原因

之前页面切换主要有几个问题：

- 一级页面切走后会丢状态
- 子页面返回时不能总是正确回到上一级
- 跨父页面跳转时体验不统一
- 滑动返回和系统返回行为不一致
- 二级页面显示时容易露出下面一级页面内容

这会导致：

- 训练页筛选条件丢失
- 列表滚动位置丢失
- 页面切换“闪一下”
- 回退逻辑不自然


### 3.2 一级页面常驻

现在四个一级页面已经改成常驻：

- 首页
- 训练一级页
- 记录一级页
- 我的一级页

实现位置：

- [Index.ets](D:/HearBridge-User/entry/src/main/ets/pages/Index.ets)

关键方法：

- `buildPersistentTopLevelPages()`

当前策略是：

- 一级页保持挂载
- 切换 tab 时不销毁
- 只做显示/隐藏切换

这样带来的直接收益：

- 滚动位置保留
- 搜索词保留
- 筛选项保留
- 当前列表结果保留
- 一级页面切换更像原生 App


### 3.3 子页面覆盖层

二级页面、详情页不再和一级页面混在同一层级裸切。

现在做法是：

- 一级页面作为底层常驻层
- 子页面作为覆盖层出现

关键方法：

- `buildAnimatedCurrentPage()`

后续又进一步调整成：

- 子页面激活时，一级页面整层直接隐藏
- 但不销毁

也就是说：

- 一级页面状态还在
- 但视觉上不会叠在下面

这一步的目的就是解决：

- “进入子页面时看到父页面很乱”
- “叠加感太重”


### 3.4 历史路由回退

现在不是简单的 if/else 写死回退，而是加了一层轻量历史栈。

关键状态与方法：

- `routeHistory`
- `captureRouteSnapshot()`
- `applyRouteSnapshot(...)`
- `pushRouteSnapshotAndTransition(...)`
- `restorePreviousRoute()`

这套逻辑支持：

- 从某个子页面跳到另一个父页面的子页面
- 返回时优先回真正的上一级
- 不会总是硬回默认父页

实际效果：

- 我的收藏 -> 训练详情 -> 返回
- 记录课程详情 -> 训练详情 -> 返回
- 我的主页 -> 账号与安全 -> 修改密码 -> 返回

这些路径现在都能按历史回退。


### 3.5 动画策略

现在切换策略已经分层：

- 四个一级页面之间切换：
  - 不加动画
- 子页面之间切换：
  - 统一轻量过渡动画

主要逻辑：

- `runNestedPageTransition(...)`

这样做的原因是：

- 一级页面切换需要干净直接
- 子页面之间切换需要层级感和连贯性


### 3.6 系统返回与滑动返回

现在整体回退顺序是：

1. 子页面优先回真正上一级
2. 回到底栏一级页面后，再滑动会先回首页
3. 只有已经在首页时，再滑一次才退出应用

并且会配合全局 Toast 给出：

- `再滑一次退出应用。`

这样整体行为更像真实 App。


## 4. 缓存优化

### 4.1 为什么只做字段缓存

这次缓存不是做资源文件缓存，而是做“接口返回字段缓存”。

原因：

- 文本和状态字段体积小
- 能显著减少重复请求
- 不会增加太多内存压力
- 图片、音视频资源本体仍然交给系统组件处理

所以这次缓存的边界是：

- 缓存文本、状态、资源 URL
- 不缓存图片字节
- 不缓存音视频字节
- 不缓存上传文件对象


### 4.2 课程详情缓存

文件：

- [CourseDetailPage.ets](D:/HearBridge-User/entry/src/main/ets/pages/CourseDetailPage.ets)

当前有两层缓存：

1. 课程详情字段缓存
   - `COURSE_DETAIL_FIELD_CACHE`
2. 课程训练会话简表缓存
   - `COURSE_SESSION_FIELD_CACHE`

作用分别是：

- 课程详情字段缓存
  - 课程名称
  - 课程介绍
  - 封面 URL
  - 标签
  - 训练类型

- 课程训练会话简表缓存
  - 训练会话 `sessionId`
  - 当前题
  - 当前题状态
  - 当前课程进度
  - 训练项简表

现在重复进入同一课程详情时：

- 先查缓存
- 命中则直接展示
- 不再重新请求课程详情和训练会话简表接口


### 4.3 训练项详情缓存

文件：

- [TrainPage.ets](D:/HearBridge-User/entry/src/main/ets/pages/TrainPage.ets)

当前也分两层：

1. 训练会话简表缓存
   - `TRAINING_SESSION_FIELD_CACHE`
2. 训练项详情字段缓存
   - `TRAINING_ITEM_DETAIL_FIELD_CACHE`

缓存边界很清楚：

- 训练会话缓存
  - 只放简表
  - 不混入单题完整详情

- 训练项详情缓存
  - 放单题的正文、说明、目标、注意点、常见错误、标签、资源 URL

这与后端接口拆分是一致的：

- 会话接口返回简表
- 单题详情接口返回完整内容


### 4.4 训练页初始化逻辑

现在训练页的初始化流程是：

1. 先查训练会话缓存
2. 命中则直接恢复：
   - `sessionId`
   - 训练项简表
   - 当前题状态
   - 当前项定位
3. 然后再看当前题详情缓存
4. 没命中单题详情时，才请求单题详情接口

收益：

- 重复进入训练页更快
- 重复打开同一训练项介绍页更快
- 当前题状态不会因为重复进入而反复重建


### 4.5 提交训练后的缓存同步

缓存不是只读的。

在提交训练成功后，前端现在会同步更新缓存中的状态：

- 当前题状态
- 下一题状态
- 当前题指向

这样下次再进入时：

- 不会用旧缓存把新状态覆盖掉
- 缓存与已完成结果保持一致


### 4.6 最近学习课程的即时本地同步

除了详情缓存，最近学习课程也做了本地即时更新。

文件：

- [Index.ets](D:/HearBridge-User/entry/src/main/ets/pages/Index.ets)
- [TrainPage.ets](D:/HearBridge-User/entry/src/main/ets/pages/TrainPage.ets)

当前逻辑：

- 训练提交成功后
- 当前课程会被立刻插到最近学习列表最前面
- 如果原来已经有同一门课，先删旧的再放最前

更新内容包括：

- 课程名
- 封面
- 进度
- 时间展示

这样就不需要等重新进入记录页才看到变化。


### 4.7 生命周期

目前这套缓存是：

- 单次应用会话内存缓存

特点：

- 应用活着时有效
- 应用被杀、重启后自然清空
- 退出登录后也会主动清空
- 修改密码重新登录时也会主动清空

这很适合当前阶段：

- 实现简单
- 风险低
- 不需要复杂持久化

## 6. 当前仍可继续优化的点

虽然今天这三块已经打通，但后续还可以继续做这些：

### 6.1 跳转动画进一步去闪

目前跨父页面子页面切换已经收敛很多，但如果后续还发现某些极端路径有闪动，可以继续升级为：

- 双页面并行动画

也就是同时保留旧页和新页过渡，而不是单层覆盖切内容。


### 6.2 收藏对象列表的本地同步再补齐

当前收藏状态本身已经能本地即时切换，但收藏对象列表还可以继续补强：

- 收藏课程后直接本地 upsert `favoriteCourses`
- 收藏训练项后直接本地 upsert `favoriteTrainingItems`


### 6.3 记录页聚合数据的静默刷新

像学习曲线、今日状态这类强聚合数据，不适合完全前端推断。

后续更合理的策略是：

- 关键动作成功后，本地即时更新轻量状态
- 同时后台静默刷新一次聚合接口
